# 深度研究.SKILL.md（拆分版 03）

## 第 5 章 输出模板、复核清单与附录工具箱

- 章节职责: 将前四章方法收束成可直接调用的输出模板与复核工具。
- 必须覆盖: 结构化回答模板、纪实式事实核查写法、自我复核清单、快速参考卡。
- 禁止越界: 不回头重写前四章理论主体，只做收束与落地。

<!-- chapter:05:start -->
第 5 章不再解释“为什么要这样研究”，而是把前四章已经建立的方法压缩成可以直接调用的工作模板（working templates）与复核工具。它服务的对象有两个：一类是已经理解前四章框架、现在要立刻开始一次研究任务的人类研究者；另一类是没有历史记忆、必须依赖显式结构执行任务的无状态 LLM。无论是哪一类使用者，第 5 章都只回答一个问题：当研究真的开始时，应该怎样稳定地产出可交付结果，而不是重新回到概念讨论。

### 5.1 输出自适应原则：让报告服务具体读者与决策场景

面向阅读的研究报告，不应先套固定大纲（fixed outline），再把问题硬塞进去；更稳妥的做法，是先判断这份文档究竟要帮谁完成什么决定，再反推结构、长度与语气。

- 先判定文档策略：优先识别主题类型（学术 / 产品 / 商业 / 技术 / 政策）、读者对象、使用场景（汇报 / 决策 / 立项 / 学习）与风险等级，再决定报告风格、结构和长度。
- 结构动态生成：禁止把固定大纲当成默认正确答案；结构应围绕用户目标最大化可用性（usability），若用户未指定，则先做轻量假设，并在开头用一句话声明当前版本可切换。
- 长度按决策成本匹配：低风险问题优先给短报告；高风险、高争议或高成本决策问题，应提供更长的论证、更完整的证据讨论与更明确的限制条件。
- 过程与交付分层，但不强制双文件：默认先交付最适合用户直接使用的主文档；只有在用户需要复核、审计或复现时，再补充过程文档（process record）或审计材料。
- 交付前一致性检查：交付前至少自检一次“风格 - 受众 - 用途”是否一致；若明显不一致，应先重写输出框架，再进入最终定稿。

如果你拿不准该写成什么样，先问自己四个问题：这份报告是谁读、读完要做什么、做错的代价有多高、对方最需要先看到什么。四个问题回答清楚后，再选模板，而不是反过来。

### 5.2 结构化回答模板：把研究结果写成可检查的交付物

一份合格的深度研究输出，不只要给结论，还要交付结论边界、支撑证据和未解决部分。下面这份参考模板（reference template）适合作为高信息密度交付的稳妥起点，字段本身不建议在未判断用途前随意删减。

```md
## 问题重述

- 研究问题: <用一句话重述原问题>
- 研究边界: <主体 / 时间 / 地点 / 版本 / 不纳入范围的部分>
- 任务目标: <本次研究要支持什么判断或行动>

## 研究路径

1. <第一轮做了什么，为什么先做这一步>
2. <第二轮新增了什么证据，策略如何调整>
3. <第三轮如何处理冲突、缺口或盲区>

## 证据要点

### 来源组 A: <原始记录 / 官方资料 / 一手材料>
- 证据 1: <事实内容>
  - 来源类型: <一手 / 二手>
  - 默认可信度: <5 / 4 / 3 / 2 / 1>
  - 在论证中的作用: <支撑 / 反驳 / 背景 / 待核>

### 来源组 B: <独立媒体 / 专业分析 / 对立观点>
- 证据 2: <事实内容>
  - 来源类型: <一手 / 二手>
  - 默认可信度: <5 / 4 / 3 / 2 / 1>
  - 在论证中的作用: <支撑 / 反驳 / 背景 / 待核>

## 推理链

1. 已确认事实: <不带推断色彩的事实>
2. 中间判断: <这些事实共同支持了什么判断>
3. 结论: <当前最稳妥的结论>
4. 关键前提: <结论成立依赖哪些前提>

## 结论确定度

- 确定度: <High / Medium / Low>
- 判断依据: <为什么只能到这个等级>

## 残余不确定

- 尚未闭合的缺口: <仍未补齐的关键点>
- 潜在盲区: <可能还没充分展开的方向>
- 若继续研究，最优先下一步: <一个最小高信息增益动作>
```

这个模板的关键，不在于“填满所有格子”，而在于强制区分四件事：问题边界、证据内容、推理过程、结论强度。结构化模板的价值，就是把这四层明确拆开。

若使用 research CLI，可以把这个模板理解成命令链的输出映射（output mapping）：`init` 对应问题重述与任务目标的起点，`node_add` 和 `graph_link` 对应研究路径与推理链，`evidence_add` / `evidence_verify` / `evidence_link` 对应证据要点，`run` 与 `graph_snapshot` 对应阶段状态，`export` 对应最终交付。一个可用的最小顺序通常是 `init -> node_add -> evidence_add -> evidence_verify -> evidence_link -> graph_snapshot -> export`。

如果研究对象是开放问题而非争议主张，模板可以稍作压缩，但不应删除“残余不确定”与“结论确定度”两项。没有这两项，输出就会滑向伪确定答案。反过来，如果研究对象是高争议、高利害或高时效问题，则应加强“研究路径”和“证据要点”两部分，明确写出版本差异、来源独立性与时间范围。

### 5.3 纪实式事实核查写法：克制叙述，不把判断伪装成事实

当输出对象是一条争议主张时，列表式模板未必最适合对外阅读。此时更稳妥的写法，是采用纪实式（documentary-style）事实核查结构：按调查顺序交代主张、溯源、核验过程、关键证据、冲突点和最后判断，让读者顺着路径看见“为什么这个结论成立”。

纪实式写法有四个要求。

第一，开头先交代被核查的最小命题，而不是先给价值判断。不要一上来写“这是假消息”或“这一说法成立”，而应先写“本次核查的对象是什么，核心争议在哪个事实上”。这样可以避免把结论偷渡进问题。

第二，正文按证据进入的顺序推进，而不是按情绪强弱推进。先写最早出现的版本、首发源头或原始记录，再写传播过程中的变化，接着写如何找到了关键原件、关键数据或独立来源，最后再解释哪些证据支持、哪些证据削弱当前主张。这样写，读者能直接看到调查路径，而不是被动接受终点判断。

第三，语气必须克制（restrained）。克制不等于冷漠，而是避免夸张、嘲讽、先验立场和语义偷换。纪实式事实核查最常见的写作失败，是用很强的措辞掩盖证据并不充分。只要证据链里仍有未闭合部分，语言就应当保留条件性，而不是用语气代替验证。

第四，结尾要明确区分“已经证实”“尚未证实”“无法证实”。事实核查不是只有真与假两档，很多时候更准确的表达是：某一部分已被原始材料证实，某一部分因版本漂移被削弱，某一部分目前缺乏足够证据，因此不能扩大解释。把这种边界交代清楚，比给出一个过于整齐的标签更重要。

可以直接采用下面这个纪实式骨架。

```md
这次核查聚焦于 <争议主张>。要判断这条说法是否成立，首先需要确认 <最小可核命题>，因为传播过程中它已经出现了 <版本变化 / 范围扩张 / 语义漂移>。

沿着传播链回溯后，可以先确认 <首发来源 / 最早版本 / 原始记录>。这一步说明 <哪部分说法确实出现过，哪部分是后续传播中被加上的>。随后再比对 <原始材料 A> 与 <独立来源 B>，可以看到二者在 <时间 / 主体 / 口径 / 数据> 上是 <一致 / 部分冲突 / 明显冲突>。

决定当前判断的关键证据是 <关键证据>。它表明 <事实层结论>。但同时仍有 <未解决限制>，因此更稳妥的写法不是“完全证实”或“完全证伪”，而是：<条件化判断>。

基于现有材料，可以确认的是 <已确认部分>；不能确认的是 <未确认部分>；如果还要进一步提高结论确定度，下一步最值得补的是 <下一步验证动作>。
```

纪实式写法适合用于事实核查、争议澄清、事件回溯、版本变迁分析等任务；不适合用于纯定义查询、简单资料汇总或低复杂度说明。写法的选择，应服务于问题结构。

### 5.4 自我复核清单：在交付前检查你有没有跳步

研究结果真正失真，往往发生在写完之后。因为一旦开始输出，人就会自然倾向于把链条补齐、把语气抬高、把未确认部分写得像已经成立。自我复核清单（self-review checklist）的作用，就是在交付前强制做最后一次反向检查。

交付前至少逐项检查以下内容。

- 问题边界是否写清楚了。读者能否一眼看懂本次研究研究了什么、没有研究什么。
- 关键结论是否都能追到证据。任一重要判断都不应只靠语气、直觉或单一转述成立。
- 支撑性证据与反驳性证据是否都被看见。若只保留支持当前观点的材料，结论可信度应自动下调。
- 来源是否足够独立。若两个来源都回到同一上游材料，就不能当作双重验证。
- 时间、主体、地点和口径是否已经对齐。若这些元信息未对齐，很多“矛盾”只是伪冲突。
- 事实、推断、观点是否分层表达。尤其要防止把解释性语言写成事实描述。
- 结论确定度是否和证据强度相匹配。证据只到中等，就不要写成高确定度。
- 残余不确定是否已经披露。没有披露限制条件的研究，通常不是真正稳健的研究。
- 下一步建议是否具体。若建议只是“继续研究”，说明当前输出仍不够结构化。

如果其中任一项无法明确回答，最安全的处理不是硬交付，而是回到前面的模板，补出那一段缺失结构。深度研究要求的是可复核，不是看起来完整。

为了提高自检效率，可以在交付前再问自己三个压缩问题。第一，我的核心结论若要被推翻，最可能被哪条反证推翻。第二，如果把这份输出交给另一个无状态 LLM，它是否能凭这份文本复现我的判断路径。第三，若读者只读“结论确定度”和“残余不确定”，是否也能理解为什么当前不能说得更满。只要这三个问题答不稳，输出就还没有完全收束。

### 5.5 从单次研究到能力沉淀：建立个人工作模板与案例库

第 5 章的最后一步，是提醒研究者把高频动作沉淀下来。深度研究的难度，并不只来自单次任务，也来自每次都从零开始。成熟研究者与新手的差异，往往不在智力，而在是否已有稳定的模板、检查点和失败经验库。

最值得沉淀的内容通常只有三类。第一类是问题拆解模板，也就是你常处理的问题类型有哪些，它们通常需要哪些子命题与信息需求。第二类是证据判断模板，包括哪些来源在你的领域里通常更可靠、哪些口径最容易误读、哪些反证路径必须常规检查。第三类是案例库（case library），记录典型成功案例、典型误判案例、常见版本漂移模式、常见伪冲突类型与常见致命缺口。

沉淀的目的，是缩短从“看见问题”到“进入研究状态”的时间。对个人研究者来说，这意味着把高频 checklist 固化下来；对无状态 LLM 来说，这意味着优先调用一套显式的最小模板。

因此，第 5 章可以被视为整份技能文档的执行面。前四章负责建立边界、设计、取证和过程控制，第 5 章则把这些能力压缩成可复制（repeatable）、可调用（callable）、可复核（reviewable）的研究工具箱。真正的落地，是在每次研究里都能稳定调用这些模板与清单。

## 附录 A 术语表

- 用途: 统一关键术语定义，避免跨章漂移。

<!-- appendix:a:start -->
| 术语 | 定义 |
| --- | --- |
| 深度研究 | 面向复杂、模糊、争议或高利害问题的循证研究活动，目标是构造可复核的判断，而不是快速给出单一答案。 |
| 普通搜索 | 以快速获取事实、定义、入口信息为主的线性检索活动，通常不承担多轮验证与置信度管理。 |
| 证据链 | 支撑某个结论的一组证据及其相互关系，强调来源、时间、作用位置与可追溯性。 |
| 推理链 | 从已确认事实到中间判断再到结论的逻辑路径，用来解释“为什么会得到这个判断”。 |
| 研究边界 | 本次研究明确纳入和排除的主体、时间、地点、版本、问题范围与任务目标。 |
| 主问题 | 本轮研究最终要回答的核心问题。 |
| 子问题 | 为回答主问题而拆解出的必要组成部分。 |
| 假设 | 当前最值得先检验的解释路径或判断方向，必须允许被证据削弱或推翻。 |
| 替代假设 | 与当前主假设竞争、修正或推翻的其他解释路径。 |
| 可证伪 | 一个判断存在潜在反证入口，理论上可以被证据推翻，而不是永远自我成立。 |
| 广搜 | 用于建立信息地图、发现主体、时间点、争议点和潜在盲区的宽范围搜索。 |
| 深搜 | 围绕单一高价值目标展开的窄而深的检索，用于打通路径断层、补关键原件或核对关键口径。 |
| 信息地图 | 对当前问题涉及的主体、时间、冲突点、空白区和已覆盖区的结构化认识。 |
| 路径断层 | 论证走到某一步时缺少关键证据，导致事实无法稳定连接到判断的断点。 |
| 来源分级 | 按默认可信起点对来源进行分层，用于决定取证顺序和验证强度，而不是决定最终真伪。 |
| 独立来源 | 不是从同一上游材料、同一爆料链或同一原始文本改写出来的来源。 |
| 一致性检验 | 对时间、地点、主体、口径、版本和机制解释逐项核对，防止伪冲突和伪一致。 |
| 事实核查 | 围绕争议主张进行溯源、核验原件、建立时间轴和搜索反证的专项流程。 |
| 知识缺口 | 已知论证链中明确缺失的证据、定义、数据或原件。 |
| 知识盲区 | 尚未进入当前问题版图、但可能显著改变结论的未知方向。 |
| 三段式反思 | 每轮动作后都要完成的状态更新协议，包括新增证据、判断变化与下一步最小动作。 |
| 停止条件 | 判断当前研究应结束的标准，通常基于证据充分性、边际增益和外部边界。 |
| 残余不确定 | 研究结束时仍未完全消除、但必须显式披露的未知部分。 |
| 致命缺口 | 一旦方向相反就足以推翻主结论或使核心证据链失效的未解决问题。 |
| 降级策略 | 在工具、权限、时间或条件不足时，将输出降为当前最佳可核实结果的处理方式。 |
| 结论确定度 | 对当前结论强度的分级表达，通常使用 High、Medium、Low 三档。 |

## 附录 B 快速参考卡

- 用途: 提供问题拆解、搜索、验证、反思、停止条件与降级的速查卡。

<!-- appendix:b:start -->
### 最小行动卡

#### 1. 启动与执行

- 启动条件: 问题复杂、信息冲突、结论高利害，且需要同时交付证据链、推理链和确定度。
- 进入前先写清楚: 研究对象、主体/时间/地点/版本边界、本轮任务目标、不纳入范围。
- 问题拆解最小模板: 主问题、子问题、主假设、替代假设、信息需求。
- 搜索切换: 先广搜看版图，再深搜补断点；若新材料只是重复转述，就退出当前路径。
- 每轮动作后: 记录新增证据、更新判断变化、给出下一步唯一优先动作。

#### 2. 收束与交付

- 证据底线: 关键结论不能只靠单一来源；共用同一上游的来源不算独立验证；关键定义、口径、版本问题优先回原文。
- 停止条件: 核心命题已被多源支撑，或连续几轮没有新增独立证据，或外部条件已阻断继续推进。
- 降级交付: 冻结已确认的证据，下调结论确定度，说明失败环节，并给出最现实的补救路径。
- 输出前三问: 关键结论能否逐条追到证据；最可能推翻它的反证是否检验过；另一个无状态 LLM 能否复现你的路径。

## 附录 C 引用与证据使用约定

- 用途: 若后续决定保留真实引文体系，在此统一说明引用格式与证据标注规则。

<!-- appendix:c:start -->
本技能采用无伪引文（clean, no fake citation）的证据使用约定。凡是进入研究输出的证据，都应满足“能指出来源、能说明作用、能交代限制”这三个最低条件。

### 最小证据规则

- 写明来源类型与来源位置；若有真实链接、文档名、数据库名、发布日期、版本号，应尽可能保留。
- 写明时间信息；高时效问题必须交代发布日期、访问时间或版本时间。
- 写明论证作用；每条证据应标记为支撑、反驳、背景或待核。
- 写明使用限制；若未见原件、来源独立性不足、口径不明，必须同时披露。
- 不得伪造编号、脚注、论文名或出处；不得把“常识”“有人说”这类模糊表述当作证据。
- 强结论需要强证据；若仍有致命缺口，结论必须条件化表达，并把缺口写入残余不确定。

### 最小合规模板

可以用下面这组字段记录关键证据，哪怕不采用正式脚注体系，也能保持可复核性。

```md
- 证据: <事实内容>
  - 来源: <机构 / 文档 / 页面 / 数据库>
  - 时间: <发布日期 / 访问时间 / 版本>
  - 类型: <原始 / 二手 / 线索>
  - 作用: <支撑 / 反驳 / 背景 / 待核>
  - 限制: <未见原件 / 独立性不足 / 口径待核 / 时效受限>
```

附录 C 的职责不是教格式排版，而是守住研究诚信（research integrity）的底线：没有真实来源，就不要伪造引用；证据还不够，就不要把判断写满；限制条件存在，就必须随结论一起交付。
